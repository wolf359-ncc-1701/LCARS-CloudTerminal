# V0.5 LCARS Grammar

V0.5 moves the UI from a themed dashboard toward a reusable LCARS element grammar.

Reference taxonomy studied from https://joernweissenborn.github.io/lcars/:

- Layout: rows, columns, fill, centered, right, bottom.
- Grid: strict unit widths and heights.
- Colors: named color classes rather than one-off values.
- Bar: horizontal fills, spacers, left/right rounded ends, decorated ends.
- Text Box: compact system labels and large readouts.
- Element: basic block/button/indicator, optionally rounded.
- Element Addition: small indicator embedded in an element.
- Element Decorator: small stripe or mark on the side of an element.
- Elbow: four corner directions.
- Bracket: decorative grouping frame.
- Audio: semantic input acknowledgement sounds.
- SVG Special Elements: use SVG for complex controls only when needed.

## How This Maps To This App

Current V0.5 implementation uses original CSS and does not import the reference framework.

Implemented now:

- Unit-based shell tokens: `--unit`, `--bar`, `--rail`.
- LCARS left rail with elbow-like top block.
- Horizontal top bars with segment caps.
- Dense element grid for left rail status blocks.
- Main display bracket/window.
- Readout text boxes.
- Telemetry rows.
- Bottom block strip.
- Red alert state.
- Synthesized semantic beep/alert sounds.

Still needed:

- Extract reusable `LcarsElement`, `LcarsBar`, `LcarsElbow`, `LcarsBracket` components.
- Add named palette classes or component props.
- Add SVG special controls for x/y pad and radial selector.
- Add better mobile grammar instead of merely stacking panels.
- Add manual info overlay that highlights element categories.

## Design Rule

Every future UI piece should be classified before implementation:

1. Is it a layout container?
2. Is it a bar?
3. Is it an element?
4. Is it a text box/readout?
5. Is it an elbow/bracket?
6. Is it a special SVG/canvas element?

If it does not fit one of these, avoid adding it until the grammar is extended intentionally.
