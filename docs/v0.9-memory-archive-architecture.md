# V0.9 Memory Archive Architecture

V0.9 upgrades the `memory` module from decorative memory diagnostics into a real LCARS archive console. The module should feel like the ship computer database access layer: file manager first, technical manual reader second, future personal cloud database bridge third.

## Source Material Ingested

Reference repository:

- `https://github.com/narwhrl/stcn-docs/tree/main`

Local clone:

- `external/stcn-docs`

TNG technical manual source:

- Markdown chapters: `external/stcn-docs/docs/docs/tng-technical-manual/*.md`
- Image assets: `external/stcn-docs/docs/assets/img/TNGTM/`
- Generated archive PDF: `output/pdf/tng-technical-manual-cn.pdf`
- Frontend-served PDF: `public/library/tng-technical-manual-cn.pdf`
- Manual manifest: `public/library/tng-manual-manifest.json`
- Project file manifest: `public/library/project-file-manifest.json`

Observed structure:

- The upstream project is a VitePress documentation site.
- The TNG manual is split into 17 chapter markdown files.
- The generated local PDF has 168 pages.
- The generated manual manifest exposes 127 outline entries with PDF page targets.
- The project file manifest currently exposes 91 local project files as a static V0.9 prototype dataset.

## Why This Matters To LCARS CloudTerminal

The manual's `computer-systems.md` is directly relevant to this project. It describes LCARS as a database access and retrieval layer, with distributed terminals, PADDs, core memory, isolinear optical storage, and near-real-time panel polling. For our project this maps neatly to:

- `Memory` mode: local archive and file retrieval console.
- Future database: file metadata, document outline, content chunks, asset relationships, and search indexes.
- Future AI assistant: semantic retrieval over manuals, project files, media, and user notes.
- Future local server: trusted file operations that the browser cannot perform alone.

## V0.9 Product Goal

When `memory` mode is active:

1. If no file is opened, the memory area becomes a full-width LCARS file manager.
2. The file manager resembles a macOS Finder-style browser in function, but not in visual style.
3. The left side shows source groups and folder trees.
4. The main area shows file collections, metadata, and available archive actions.
5. The right rail mode action blocks become real file/archive actions instead of decorative labels.
6. When a file is opened, the file manager compresses into a left directory tree and the right side becomes a reader.
7. PDF and markdown outlines appear in the tree and can jump the reader to a section.

## State Model

Frontend state should be small and explicit.

```ts
type MemorySource = "manual" | "project";
type MemoryLayout = "browser" | "reader";

interface MemoryArchiveState {
  source: MemorySource;
  layout: MemoryLayout;
  selectedNodeId: string | null;
  openedItemId: string | null;
  activePage: number | null;
  query: string;
  treeExpanded: Record<string, boolean>;
}
```

Layout rules:

- `layout = "browser"` when `openedItemId` is null.
- `layout = "reader"` when a manual section, PDF, markdown file, or project file is opened.
- Closing the reader resets `openedItemId` and returns to the full file browser.
- Switching source keeps the layout if possible, but should close incompatible open items.

### V1 Backend Target

The browser cannot safely read arbitrary local files by itself. V1 should add a local trusted API:

```text
GET  /api/files/tree?root=workspace
GET  /api/files/read?path=...
GET  /api/files/meta?path=...
GET  /api/archive/manuals
GET  /api/archive/manuals/:id/outline
GET  /api/archive/manuals/:id/page/:page
POST /api/search
```

Security rules:

- Only serve files under allowlisted workspace roots.
- Never allow `..` traversal.
- Expose write/delete only after a separate explicit V1 task.
- V0.9 is read-only.

## Interaction Requirements

File manager mode:

- Click a manual chapter tile: open PDF reader at that chapter page.
- Click `TNG Manual`: source becomes `manual`.
- Click `Project Files`: source becomes `project`.
- Click a project file row:
  - if readable, open reader view and show text preview if available later;
  - if binary, open metadata panel and show "binary preview pending".
- Search filters visible items by file name, title, extension, and chapter title.

Reader mode:

- Left tree remains visible.
- Manual outline nodes jump the PDF iframe to `#page=N`.
- Project file tree nodes switch selected file.
- Close button returns to full manager mode.
- The transition from browser to reader compresses the manager into the tree rather than replacing the whole layout abruptly.

Right rail actions when `memory` is active:

- `MANUAL ARCHIVE`: switch to TNG manual source.
- `PROJECT FILES`: switch to project file source.
- `OPEN INDEX`: open the first manual chapter or root project index.
- `CLOSE READER`: visible/enabled only when a file is open.
- `FILTER RESET`: clear query and selection.

These replace meaningless placeholder blocks. They must be real buttons wired to state.

## Visual Rules

Follow the established LCARS design grammar:

- Keep the current V0.78 top-left elbow contract.
- Keep V0.79 viewport containment.
- Do not reintroduce Quick Actions.
- Do not add soft generic cards inside cards.
- Use LCARS rail/band/bracket grammar:
  - left tree can be a dense vertical LCARS rail,
  - file rows should be rectangular LCARS list strips,
  - open reader uses one large framed viewport.
- Use the Microgramma font for labels, buttons, headings, file names, and readout values.
- Body text in the reader may use a readable system font if needed, but metadata and chrome stay LCARS.
- Avoid same-color text/background collisions, especially cyan on cyan and gray on gray.
- Respect `prefers-reduced-motion`.

## Animation Rules

Motion should be operational, not decorative.

- Browser to reader:
  - collection grid fades/slides left into a narrow tree pane;
  - reader pane expands from right with a short LCARS scan line;
  - use existing `panel-enter`, `lcars-scan`, and soft status-dot animation patterns where possible.
- Reader to browser:
  - reader fades out;
  - tree expands back to collection grid;
  - no full page jump.
- Source switch:
  - use a short segmented wipe or row-scan transition.
- File open:
  - play existing `archive` or `confirm` audio through `useLcars().beep`.

## Acceptance Criteria For Gemini

- `MemoryView` is no longer a decorative hardware-sector mock.
- No file open: file manager occupies the full memory content area.
- File open: left tree plus right reader layout.
- Right rail actions perform Memory-specific actions when Memory mode is active.
- TNG manual can be opened in the reader from `/library/tng-technical-manual-cn.pdf`.
- Manual outline uses `/library/tng-manual-manifest.json`.
- Project file manager uses `/library/project-file-manifest.json`.
- `npm run build` passes.
- No regressions to Bridge, Habitat, Power, Actions, top-left elbow, or viewport containment.
